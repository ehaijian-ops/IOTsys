## 一、技术选型

## 三、协议说明

### 3.1 AP3000 电单车充电桩协议 (V8.6)

**通信方式**: TCP Socket 长连接

**帧格式** (二进制，小端模式):
```
帧头(DNY) | 长度(2B) | 设备ID(4B) | 消息ID(2B) | 命令(1B) | 数据(nB) | 校验(2B)
3 字节    | 2 字节   | 4 字节    | 2 字节    | 1 字节   | n 字节   | 2 字节
```

- **帧头**: `DNY` (0x44 0x4E 0x59)
- **长度**: 命令+数据+设备ID(4)+消息ID(2) 总字节数, ≤256
- **设备ID**: 4字节小端，从二维码MAC地址反算（标识码+设备号）
  - 标识码: 0x03=单路, 0x04=双路, 0x05=10路, 0x06=16路, 0x07=12路
- **消息ID**: 每包不同，防重放，响应时回复相同ID
- **校验**: 除帧头外所有字节累加和取低16位
- **重发机制**: 默认发1次，超时15秒重发，包间间隔≥0.5秒

**核心指令表**:

| 命令 | 方向 | 说明 |
|------|------|------|
| 0x01 | 设备→服务器 | 设备登录(旧版，含端口状态/功率) |
| 0x02 | 设备→服务器 | 刷卡/余额查询 |
| 0x03 | 设备→服务器 | 充电记录上传(时长/电量/停止原因) |
| 0x04 | 设备→服务器 | 端口充电确认 |
| 0x06 | 设备→服务器 | 充电中实时数据(每5秒) |
| 0x20 | 设备→服务器 | 设备注册(新版，含固件版本/工作模式) |
| 0x21 | 设备→服务器 | 设备状态上报(电压/端口状态/信号) |
| 0x22 | 设备→服务器 | 获取服务器时间 |
| 0x81 | 服务器→设备 | 查询设备状态 |
| 0x82 | 服务器→设备 | 启动/停止充电(含模式/金额/时长) |
| 0x83 | 服务器→设备 | 设置运行参数1.1(拔枪识别等) |
| 0x84 | 服务器→设备 | 设置运行参数1.2(浮充/温度保护等) |
| 0x85 | 服务器→设备 | 设置最大充电时长/功率/过欠压 |
| 0x86 | 服务器→设备 | 设置用户密码 |
| 0x87 | 服务器→设备 | 远程重启 |
| 0x88 | 服务器→设备 | 清除存储满 |
| 0x89 | 服务器→设备 | 语音播报 |
| 0x8A | 服务器→设备 | 修改充电模式/时长/金额 |
| 0x8D | 服务器→设备 | 设置工作模式(投币/刷卡) |
| 0x8F | 服务器→设备 | 设置TC刷卡模式 |
| 0xE0/E4/F8 | 服务器→设备 | OTA固件升级 |

**端口状态码**:
- 0x00=空闲, 0x01=充电中, 0x02=已扫码未充电, 0x03=已扫码已充电
- 0x04=短路, 0x05=浮充, 0x06=存储满坏, 0x08=接触器/空开端
- 0x09=继电器粘连, 0x0B=继电器无法闭合, 0x0D=短路预警

**停止原因码**:
- 0=正常充满, 1=手动停止, 2=达到最大时间, 3=达到预设时间, 4=达到预设电量
- 5=用户拔枪, 6=过载保护, 7=远程停止, 0x0A=环境温度过高, 0x10=水浸断电
- 0x19=存储故障, 0x1A=过压, 0x1B=欠压, 0x1C=低功率断电

### 3.2 TF100 电动汽车充电桩协议 (V2.9)

**通信方式**: TCP Socket 长连接

**帧格式** (JSON文本):
```
CCMD:长度{JSON}   ← 设备上报
SCMD:长度{JSON}   ← 服务器下发
```

- 长度字段: 3位数字(不足前补0), 长度≥1000时用4位
- **设备ID**: `devId` 字段, 数字型 (如 13001001)
- **消息ID**: `msgId` 字段, 防重放, 响应时回复相同ID
- **指令类型**: `msgType` 字段标识

**核心指令表**:

| 20 | 设备→服务器 | 设备注册(含固件版本/端口数/TC模式) |

| 23 | 设备→服务器 | 推送信息(枪状态变更) |
| 24 | 设备→服务器 | 地锁心跳 |

**端口状态码**:
- 0=空闲, 1=充电中, 2=等待充电确认, 3=已充满, 4=定时预约, 5=已插枪未充电

**停止原因码**:
- 0=手动停止, 1=达到最大时间, 2=达到预设时间, 3=达到预设电量, 4=达到预设金额

**告警类型 (msgType=4)**:
- 0=正常, 1=存储损坏, 2=接地故障, 3=继电器粘连, 4=漏电故障, 5=CP电压异常
- 6=急停, 7=漏电, 8=短路, 9=门锁, 10=输入电压异常, 11=继电器脱扣, 12=桩停机

                    ▼

              ┌─────┴─────┐
              ▼           ▼

              │           │
              │           ▼
              │    ┌──────────────┐
              │    │ 告警判断      │
              │    │ (规则引擎)    │
              │    └──────┬───────┘
              │           │ 触发告警

              │    ┌──────────────┐

### 4.2 指令下发流程

```
前端 ──HTTP POST──▶ Command Service
                        │
                   写入 MySQL (status=待下发)
                        │
                   协议适配器 Encode() (根据设备协议)
                        │
                   AP3000: 构建二进制帧(DNY+累加和)
                   TF100:  构建SCMD JSON帧
                        │
                   TCP 发送到设备
                        │
                        ▼
                      设备
                        │
                  执行并返回结果
                        │
                        ▼
                  协议适配器 DecodeResponse()
                        │
                   更新 MySQL 指令状态
                        │
                   发送到 Kafka (device.command.result)
                        │
                        ▼
                   WebSocket 推送结果给前端
```

## 五、协议适配器扩展机制

### 5.1 适配器接口定义

```go
type ProtocolAdapter interface {
    Name() string                                    // 协议标识 (如 AP3000_v2)
    Version() string                                 // 协议版本
    DeviceType() string                              // 设备类型 (ebike_charger/ev_charger)
    Decode(raw []byte) (*StandardData, error)        // 解析设备上报 → 标准模型
    Encode(cmd *StandardCommand) ([]byte, error)     // 标准指令 → 设备协议帧
    DecodeResponse(raw []byte) (*StandardCommandResponse, error) // 解析指令响应
    Validate(raw []byte) bool                        // 校验报文合法性
}
```

### 5.2 新增协议步骤

1. 在 `internal/protocol/adapters/` 下创建新目录（如 `ocpp16/`）
2. 实现 `ProtocolAdapter` 接口的所有方法
3. 在 `init()` 中调用 `engine.Register()` 注册适配器
4. 添加协议配置到数据库 `protocols` 表

**示例**: 新增 OCPP 1.6J 协议支持
```go
// internal/protocol/adapters/ocpp16/ocpp16.go
package ocpp16

func init() {
    engine.Register(&OCPP16Adapter{})
}

type OCPP16Adapter struct{}

func (a *OCPP16Adapter) Name() string { return "OCPP16_v1" }
func (a *OCPP16Adapter) Version() string { return "1.6" }
func (a *OCPP16Adapter) DeviceType() string { return "ev_charger" }
// ... 实现其他方法
```

### 5.3 协议自动检测

TCP Server 在首次接收设备数据时，通过 `engine.DetectProtocol()` 遍历所有已注册适配器的 `Validate()` 方法，自动识别设备协议类型：

- **AP3000**: 检测帧头 `0x44 0x4E 0x59` (DNY)
- **TF100**: 检测前缀 `CCMD:` 或 `SCMD:`
