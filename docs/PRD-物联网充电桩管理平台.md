## 一、产品概述

### 1.1 背景

随着电动自行车和新能源汽车的普及，充电桩的数量快速增长。目前存在多家设备厂商（如 AP3000 电单车充电桩、TF100 汽车充电桩），各有独立的通信协议和管理系统。运营商需要一个统一的平台来管理多品牌、多类型的充电桩设备，实现：

- 设备数据统一采集（电压、电流、功率、温度、状态等）
- 远程指令下发（启停充电、参数配置、OTA升级等）
- 跨厂商协议适配，方便后续新增设备类型
- 支撑超过 5 万台设备的并发接入

### 1.2 产品目标

| 目标 | 衡量指标 |
|------|---------|
| 统一设备接入 | 支持 2+ 种充电桩协议，新增协议 < 3 天完成适配 |
| 高并发支撑 | 单集群支持 5 万+ 设备在线，消息处理延迟 < 200ms |
| 设备远程管控 | 指令下发成功率 > 99.5%，响应时间 < 3s |
| 实时监控告警 | 设备异常 30s 内推送告警 |
| 开放扩展 | 提供标准 API，支持第三方系统集成 |

### 1.3 目标用户

| 角色 | 描述 | 使用场景 | 平台 |
|------|------|---------|------|
| **运维工程师** | 负责充电桩日常巡检和维护 | 移动端：扫码查看设备状态、处理告警 | 移动端 |
| **运营管理员** | 负责充电站运营、定价策略 | PC端：设备管理、费率配置、数据分析 | PC端 |
| **系统管理员** | 负责平台配置、厂商协议管理 | PC端：协议适配、权限配置、系统监控 | PC端 |
| **终端用户（车主）** | 使用充电桩充电的消费者 | 移动端：扫码充电、查看充电进度、支付 | 移动端（H5/小程序） |
| **设备厂商** | 提供充电桩硬件 | 提供协议文档，配合联调 | 文档 |

### 1.4 成功标准

- [ ] 平台支持 AP3000 电单车充电桩、TF100 汽车充电桩的完整接入
- [ ] 单集群 5 万设备并发在线，CPU < 60%，内存 < 16GB
- [ ] 指令下发端到端延迟 P99 < 3s
- [ ] 新增一种设备协议，开发工作量 < 3 人天
- [ ] 平台核心功能可用性 > 99.9%

### 2.2 模块一：设备接入

#### 2.2.1 设备连接管理

**用户故事：**
```
作为 系统管理员
我想要 查看所有设备的在线状态和连接信息
以便 快速定位离线设备并排查问题
```

**功能描述：**
- 支持 MQTT/TCP/WebSocket 多协议设备连接
- 设备通过唯一标识（设备SN/MAC）进行认证接入
- 连接状态实时同步至 Redis，前端通过 WebSocket 获取状态变更

**核心流程：**
```
设备上电 → 建立 TCP/MQTT 连接 → 发送注册报文
→ 协议适配器解析 → 设备认证 → 建立会话 → 开始数据上报
```

---

#### 2.2.2 协议适配引擎

**用户故事：**
```
作为 系统管理员
我想要 通过配置即可接入新厂商的设备协议
以便 不需要修改核心代码就能支持新设备
```

**功能描述：**
- 插件化协议适配架构，每种设备协议作为一个独立适配器
- 协议解析器负责：帧解析 → 数据提取 → 标准化转换 → 事件发布
- 支持通过配置注册新协议（协议标识、帧格式、字段映射）

**协议适配流程：**
```
原始报文 → 帧解析器(解帧/校验) → 字段映射器(厂商字段→标准模型) 
→ 数据标准化 → 发布到 MQ → 存储&业务处理
```

**当前支持协议：**
| 协议标识 | 设备类型 | 厂商 | 通信方式 |
|---------|---------|------|---------|
| AP3000_v2 | 电单车充电桩 | - | TCP 长连接 |
| TF100_v1 | 电动汽车充电桩 | - | TCP 长连接 |

### 2.3 模块二：设备管理

#### 2.3.1 设备台账

**功能描述：**
- 设备 CRUD（创建、查看、编辑、删除/停用）
- 设备信息：SN、设备类型、厂商、协议、所属站点、安装位置、固件版本、激活时间

#### 2.3.2 设备分组

**功能描述：**
- 按站点/区域/类型对设备分组管理
- 支持批量操作（批量重启、批量参数下发）

### 2.4 模块三：实时监控

#### 2.4.1 设备实时状态

**功能描述：**
- 设备在线/离线/故障 状态实时展示
- 实时数据：电压、电流、功率、温度、充电进度、累计电量
- 地图视图展示设备分布

**信息展示：**

| 字段 | 数据来源 | 展示形式 |
|------|---------|---------|
| 设备状态 | Redis 缓存 | 绿/灰/红 状态灯 |
| 实时电压/电流 | 设备上报→Redis→WebSocket推送 | 数字+仪表盘 |
| 充电进度 | 设备上报 | 进度条+百分比 |
| 设备温度 | 设备上报 | 数字+温度计图标 |
| 今日充电量 | Redis 累加 | 数字 |

### 2.5 模块四：指令下发

#### 2.5.1 远程启停

**用户故事：**
```
作为 运营管理员
我想要 远程控制充电桩的启停
以便 处理异常充电或紧急情况
```

**功能描述：**
- 支持单设备/批量远程启动充电、停止充电
- 指令通过 MQ 异步下发，保证可靠性
- 指令状态追踪：待下发 → 已下发 → 设备响应 → 执行成功/失败/超时

**指令生命周期：**
```
创建指令 → 写入指令表(status=待下发) → 发送MQ 
→ 协议适配器接收 → 编码为设备协议帧 → 发送至设备
→ 设备响应 → 更新指令状态 → 通知前端
```

## 三、非功能需求

### 3.1 性能要求

| 指标 | 要求 |
|------|------|
| 设备连接并发 | ≥ 50,000 长连接 |
| 消息处理吞吐 | ≥ 10,000 msg/s |
| 数据上报延迟 | P99 < 200ms |
| 指令下发延迟 | P99 < 3s |
| 管理端 API 响应 | P99 < 500ms |
| 页面加载时间 | ≤ 2s |

### 3.2 可用性要求

- 核心服务（设备接入、指令下发）可用性 > 99.9%
- 支持服务无感重启（优雅关闭 + 连接保持）
- 设备断线自动重连，数据不丢失

### 3.3 安全要求

- [ ] 设备认证（Token/证书）
- [ ] 通信加密（TLS）
- [ ] API 鉴权（JWT + RBAC）
- [ ] 操作日志审计
- [ ] SQL 注入/XSS/CSRF 防护
- [ ] 敏感数据加密存储

### 3.4 扩展性要求

- [ ] 协议插件化，新增协议无需修改核心代码
- [ ] 水平扩展支持（无状态服务 + 分布式缓存）
- [ ] 开放 REST API，支持第三方对接
